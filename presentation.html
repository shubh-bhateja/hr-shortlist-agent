📌 The Problem

HR teams screen hundreds of applications per role, leading to:

  • Fatigue & Inconsistency — Reviewer #1 at 9 AM ≠ Reviewer #1 at 5 PM
  • Unconscious Bias — Name, university, and format influence decisions
  • No Transparency — "Why was this candidate rejected?" has no auditable answer
  • Time Drain — 6–8 minutes per resume × 200 applicants = 25+ hours per role

Goal: Build an AI agent that standardises evaluation, surfaces best-fit candidates, human in the loop for final decisions.

💡 Solution Overview

What the agent does, end-to-end:

Inputs

  • 📋 Job Description (free text)
  • 📁 Resumes (PDF / DOCX)
  • 🔗 LinkedIn profiles (JSON export)

Outputs

  • 📊 Ranked shortlist with scores
  • 📄 Professional PDF report
  • ✏️ HR override log (audit trail)
  • 📥 JSON export for ATS integration

📋 JD Upload → 🔍 Parse → 📐 Score → 📊 Rank → 📄 Report → ✏️ Override

🏗️ Architecture — LangGraph Multi-Agent Pipeline

Each node is a pure function with Pydantic-validated I/O. State flows through the graph automatically.

START → Node 1
JD Parser
LLM → JobRequirements → Node 2
Profile Parser
PyMuPDF + LLM + PII Mask → Node 3
Scorer
Embeddings + LLM Rubric → Node 4
Report Gen
ReportLab PDF → END
Why LangGraph over a simple LangChain chain?
  • Explicit DAG with typed state — each node independently testable
  • State inspectable at every step (debuggable)
  • Extensible: add conditional edges (e.g., skip scoring if JD parse fails) without rewriting
  • Native LangSmith integration — every node traced automatically

📐 Dual-Mode Scoring Engine

The key differentiator — combining quantitative embeddings with qualitative LLM reasoning.

Mode 1: Semantic Similarity

  • SentenceTransformer encodes JD & candidate
  • Cosine similarity → 0.0 to 1.0 score
  • Quantitative, deterministic signal
  • Runs locally — no API call needed

Mode 2: LLM Rubric Reasoning

  • Llama 3.3 scores 5 dimensions (0-10)
  • JSON mode + Pydantic validation
  • Receives similarity as calibration anchor
  • One-sentence justification per dimension

✍️ Prompt Engineering Journey

5 iterations from naive to production-grade — showing the thought process.

VersionChangeResult
v1Single prompt for JD + scoringInconsistent dimension coverage, schema drift
v2Separated JD parser from scorerBetter schema compliance, but inflated scores
v3Added cosine similarity as contextScores better calibrated to semantic reality
v4Explicit threshold table for recommendationsConsistent labelling (STRONG HIRE → NO HIRE)
v5 (final)One-sentence justification constraint + negative anchoringProduction-ready, useful HR output

Key Prompt Strategies:
  • Domain framing: "You are a senior HR analyst" — reduces hallucination
  • Negative constraint: "Reserve 9-10 for exceptional evidence, not assumptions"
  • Data isolation: Resume text in delimited blocks, not mixed with instructions
  • Anchor signal: Cosine similarity passed explicitly for calibration

🎓 Key Learnings & Future Work

What I Learned

  • Structured output (Pydantic + JSON mode) eliminates 90% of parsing headaches
  • Dual-mode scoring prevents both LLM hallucination and pure-embedding blindness
  • PII masking must happen before the API call — not after
  • LangGraph's explicit state model makes debugging vastly easier than chains
  • SQLite caching saves 70%+ API costs during iterative development

Future Improvements

  • On-premise LLM (Ollama + LLaMA 3) for full PII containment
  • Batch parallel processing for 100+ resumes
  • ATS webhook integration (Greenhouse, Lever)
  • Bias audit: statistical analysis across demographics
  • Multi-language resume support
  • Live LinkedIn scraping via RapidAPI
